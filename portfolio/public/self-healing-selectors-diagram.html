Self-Healing Selectors — How it works

Every capture runs the same path. Two checks decide a step's fate: was the element found, and — driven by the product's Heal mode — should it heal (sometimes even when the first selector worked). An opaque stepRef rides the whole way so a heal lands back on the exact recipe step.

Heal mode · per product

Off nothing changes (default)
On-fallback heal when a backup is used
Always refresh every run — best for audits

Key rules

Runs only on headless paths (capture, audit)
At least one selector must still resolve
One save operation; every healed selector validated

Two orchestrators, one flow. The App runs this for product captures; the Audit Lambda runs it for scheduled audits — headless, with no app or browser session, reading the recipe straight from the database. Both use the same build-and-save code above, so a heal looks identical either way. (The interactive CloudBrowser editing session is the one path excluded — we never rewrite selectors under a live user.)

Why stepRef? Selectors get flattened and re-ordered between the orchestrator, the API and Puppetshot — so position can't identify a step. We stamp each step with its StepAction id and echo it back in healed[ ], so the save targets the exact right step. Audits run on a cadence, keeping every still-resolving selector fresh — recipes never reach the “all-dead” state.

Proof of concept — a real run

This is the healing engine (Blocks 0–3, already built) running for real inside a locally-built Puppetshot, driven directly over HTTP. The step's stored selectors have mostly rotted — a dead id, a dead class, and a text selector that no longer exists — but the element is still reachable through one surviving fallback (text/About me), so the engine regenerates a fresh, validated set.

How it was run
  1. Built the selector engine into an injectable bundle — npm run build:selector-computer in browser-tools produces selector-computer.iife.js, vendored into Puppetshot. This is the artifact Puppetshot injects into the page to recompute selectors (it's the engine our recorder already uses).
  2. Built Puppetshot locally and started it as an HTTP server: ENABLE_PUPPETSHOT_SERVER=true PORT=3001 node ./dist/src/index.js
  3. Sent a real capture request to POST localhost:3001/screenshot/sync (via the scripts/heal-test.sh helper)
  4. Used a step with deliberately broken selectors plus one working fallback, pointed at the live site micheldpcarlos.com — simulating a recipe whose selectors have rotted
  5. Set healMode: on-fallback and inspected the returned healed[ ]

Where this sits in the flow above: it exercises the Puppetshot box (steps ③–⑥) directly — replay → recompute → validate → report healed[ ]. The orchestrator and save-back steps are stood in for by the curl call here; wiring the API relay and the applyHealedSelectors save is the remaining work (the next blocks).

POST /screenshot/sync
{
  "healMode": "on-fallback",
  "steps": [
    { "type": "navigate", "url": "https://micheldpcarlos.com" },
    {
      "type": "click",
      "stepRef": "stepaction-abc-123",
      "selectors": [
        "#broken_primary",      // dead
        ".cascawc",             // dead
        "text/Nothing Here",    // dead
        "text/About me"         // still resolves ✓
      ]
    }
  ]
}
Response HTTP 202
"healed": [
  {
    "stepRef": "stepaction-abc-123",
    "reason": "fallback",
    "winningSelector": "text/About me",
    "healed": [
      "aria/About me",
      "a.VPButton",
      "xpath///*[@id=\"VPContent\"]/div/div[1]/div/div[1]/div/div/a",
      "pierce/a.VPButton",
      "text/About me"
    ]
  }
]
What it shows: a four-selector list with only one survivor heals back into a fresh, complete set — aria/ · css · xpath/ · pierce/ · text/ — each validated to resolve to that same element, reported under the original step via its stepRef so it saves back to exactly the right place.

All selector-bearing targets now heal. Hover, the crop element, and annotations were wired up after the battery surfaced them — so click · hover · crop · annotations · and the atomic change/waitForElement steps all heal. For the crop element and annotations the orchestrator stamps a stepRef (elementSelectorStepRef / per-annotation) so the heal maps back to the right step. The remaining work is purely downstream: carrying healed[ ] through the API and the applyHealedSelectors save.
